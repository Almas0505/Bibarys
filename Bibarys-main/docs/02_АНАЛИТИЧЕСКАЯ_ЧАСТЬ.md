# ЧАСТЬ 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

## 1.1. Современное состояние автоматизации торговых предприятий

### 1.1.1. Характеристика торговых процессов в продуктовом магазине

Деятельность продуктового магазина представляет собой сложный комплекс взаимосвязанных бизнес-процессов, требующих тщательного контроля и учета. Рассмотрим основные этапы торгового цикла:

**1. Прием товара от поставщиков**

На этом этапе осуществляется:
- Получение накладных от поставщиков с перечнем товаров, количеством и ценами;
- Физическая приемка товара с проверкой количества, качества и сроков годности;
- Сверка фактически полученного товара с данными в накладной;
- Оформление приходных документов;
- Внесение информации о новых поступлениях в учетную систему.

**Проблемы традиционного подхода:**
- Ручное заполнение приходных документов занимает 30-40 минут на каждую поставку;
- Высокий риск ошибок при подсчете большого количества позиций;
- Сложность контроля сроков годности при большом ассортименте;
- Отсутствие оперативной информации о поступлениях для планирования продаж.

**2. Складирование и хранение товаров**

Включает в себя:
- Размещение товара на складе или в торговом зале с учетом условий хранения;
- Поддержание оптимальных остатков товаров;
- Контроль сроков годности и ротация товара по принципу FIFO (First In, First Out);
- Проведение периодической инвентаризации для выявления недостач или излишков;
- Списание просроченных или поврежденных товаров.

**Проблемы традиционного подхода:**
- Отсутствие оперативной информации об остатках приводит к дефициту популярных товаров;
- Затоваривание медленно продаваемыми позициями замораживает оборотные средства;
- Пропуск сроков годности ведет к финансовым потерям при списании;
- Инвентаризация требует закрытия магазина и занимает целый день работы.

**3. Продажа товаров покупателям**

Процесс продажи включает:
- Консультирование покупателей о наличии и характеристиках товаров;
- Формирование корзины покупателя (выбор товаров);
- Расчет общей стоимости покупки с учетом скидок и акций;
- Проведение оплаты (наличными или картой);
- Выдача чека покупателю;
- Списание проданных товаров из остатков.

**Проблемы традиционного подхода:**
- Продавец не может быстро проверить наличие товара на складе;
- Ручной расчет стоимости увеличивает время обслуживания и создает очереди;
- Ошибки в подсчетах приводят к конфликтным ситуациям;
- Отсутствие учета предпочтений постоянных клиентов.

**4. Формирование отчетности**

Завершающий этап торгового цикла:
- Подсчет выручки за день/неделю/месяц;
- Сверка фактической выручки с остатками товара;
- Анализ продаж по категориям и отдельным товарам;
- Расчет прибыли и рентабельности;
- Планирование закупок на основе динамики продаж;
- Подготовка налоговой и бухгалтерской отчетности.

**Проблемы традиционного подхода:**
- Формирование отчетов вручную занимает 2-3 часа ежедневно;
- Аналитика запаздывает на 3-5 дней, что снижает оперативность управленческих решений;
- Невозможность построения графиков и визуализации данных;
- Сложность выявления трендов и закономерностей в продажах.

### 1.1.2. Необходимость автоматизации учетных процессов

Анализ проблем традиционного ведения торговли показывает критическую необходимость внедрения автоматизированных систем учета. По данным исследований, магазины, использующие автоматизированный учет, имеют следующие преимущества:

- **Сокращение времени на учетные операции на 60-70%** - автоматический расчет остатков, формирование отчетов одним нажатием кнопки;
- **Снижение ошибок учета на 85-90%** - исключение ручного ввода и подсчетов;
- **Увеличение оборачиваемости товаров на 15-20%** - благодаря оперативной информации об остатках и динамике продаж;
- **Сокращение потерь от просрочки на 30-40%** - за счет автоматического контроля сроков годности;
- **Повышение удовлетворенности клиентов** - сокращение времени обслуживания, отсутствие ошибок в расчетах.

### 1.1.3. Современные тенденции в автоматизации торговли

На современном этапе развития информационных технологий в сфере торговли наблюдаются следующие тенденции:

**1. Переход к веб-технологиям**
- Отказ от настольных приложений в пользу веб-интерфейсов;
- Доступность системы с любого устройства через браузер;
- Автоматическое обновление системы без установки на клиентских компьютерах.

**2. Облачные решения**
- Хранение данных в облаке обеспечивает доступ из любой точки мира;
- Снижение затрат на покупку и обслуживание серверного оборудования;
- Автоматическое резервное копирование данных.

**3. Мобильность**
- Управление магазином с мобильных устройств (смартфонов, планшетов);
- Мобильные приложения для приема заказов и работы со складом;
- Push-уведомления о критических событиях (низкие остатки, новые заказы).

**4. Интеграция с онлайн-продажами**
- Единая система учета для офлайн и онлайн каналов продаж;
- Синхронизация остатков между физическим магазином и интернет-магазином;
- Возможность онлайн-заказа с самовывозом из магазина.

**5. Аналитика и бизнес-интеллект**
- Визуализация данных о продажах в виде графиков и диаграмм;
- Прогнозирование спроса на основе исторических данных;
- Сегментация клиентов и персонализация предложений.

## 1.2. Обзор аналогичных систем (анализ аналогов)

Для обоснования выбора подхода к разработке системы автоматизации товарного учета проведем сравнительный анализ существующих решений на рынке.

### 1.2.1. Microsoft Excel и Google Sheets

**Описание:**
Табличные процессоры широко используются малыми торговыми предприятиями для ведения учета товаров. Учет ведется в виде таблиц с колонками: название товара, категория, количество, цена, сумма.

**Преимущества:**

1. **Доступность и бесплатность**
   - Microsoft Excel входит в состав офисного пакета, установленного на большинстве компьютеров;
   - Google Sheets полностью бесплатен и работает в браузере;
   - Не требуется дополнительных инвестиций в программное обеспечение.

2. **Простота использования**
   - Интуитивно понятный интерфейс, знакомый большинству пользователей;
   - Не требуется специального обучения персонала;
   - Возможность быстрого старта без привлечения IT-специалистов.

3. **Гибкость настройки**
   - Пользователь сам создает структуру таблицы под свои нужды;
   - Легко добавлять новые колонки и изменять формулы;
   - Возможность создания макросов для автоматизации рутинных операций.

**Недостатки:**

1. **Неспособность работы с большими объемами данных**
   - При накоплении десятков тысяч строк Excel начинает работать медленно;
   - Ограничение на размер файла (до 2 ГБ);
   - Долгое открытие и сохранение больших файлов.

2. **Отсутствие многопользовательского доступа**
   - Одновременная работа нескольких пользователей приводит к конфликтам версий;
   - Риск потери данных при одновременном редактировании;
   - Google Sheets решает эту проблему, но имеет другие ограничения.

3. **Низкий уровень безопасности**
   - Файлы Excel легко копируются и передаются третьим лицам;
   - Отсутствие разграничения прав доступа к данным;
   - Пароли на файлы легко взламываются;
   - Нет логирования действий пользователей.

4. **Высокий риск ошибок**
   - Пользователь может случайно удалить формулы или важные данные;
   - Ошибки в формулах приводят к неверным расчетам;
   - Отсутствие валидации вводимых данных.

5. **Отсутствие автоматизации бизнес-процессов**
   - Все операции выполняются вручную;
   - Нет автоматического контроля остатков и сроков годности;
   - Формирование отчетов требует ручной работы.

**Вывод:**
Microsoft Excel подходит только для очень малых магазинов с ограниченным ассортиментом (до 100 позиций) и небольшим количеством операций. При росте бизнеса необходим переход на специализированные системы.

### 1.2.2. 1С: Предприятие. Управление торговлей

**Описание:**
1С: Предприятие является лидером рынка автоматизации торговли в России и странах СНГ. Конфигурация "Управление торговлей" предназначена для комплексной автоматизации торговых предприятий.

**Преимущества:**

1. **Функциональная мощность**
   - Полный цикл автоматизации: от закупок до продаж и отчетности;
   - Поддержка множества складов и торговых точек;
   - Управление взаимоотношениями с клиентами (CRM);
   - Интеграция с банками, онлайн-кассами, оборудованием.

2. **Соответствие законодательству**
   - Автоматическое формирование всех необходимых отчетов для налоговых органов;
   - Регулярные обновления при изменении законодательства;
   - Поддержка ЕГАИС (для алкогольной продукции), маркировки товаров.

3. **Надежность и масштабируемость**
   - Проверенное годами решение, используемое тысячами предприятий;
   - Способность обрабатывать миллионы документов;
   - Возможность роста системы вместе с бизнесом.

4. **Развитая экосистема**
   - Большое количество специалистов по 1С на рынке труда;
   - Множество дополнительных модулей и расширений;
   - Активное комьюнити и форумы для решения проблем.

**Недостатки:**

1. **Высокая стоимость**
   - Стоимость лицензии на 1 пользователя: от 13 000 до 30 000 тенге;
   - Дополнительные расходы на сервер и инфраструктуру: от 200 000 тенге;
   - Ежегодная оплата технической поддержки: 20-30% от стоимости лицензий;
   - **Общие затраты на внедрение для малого магазина: 500 000 - 1 000 000 тенге**.

2. **Сложность настройки и использования**
   - Необходимость привлечения специалиста-программиста 1С для внедрения;
   - Стоимость услуг программиста 1С: от 5 000 тенге/час;
   - Длительный период обучения персонала (2-4 недели);
   - Сложный интерфейс, перегруженный функциями.

3. **Избыточная функциональность**
   - 80-90% возможностей системы не используются в малом бизнесе;
   - Платите за функции, которые не нужны вашему магазину;
   - Перегруженный интерфейс затрудняет работу.

4. **Зависимость от поставщика**
   - Привязка к экосистеме 1С, сложность миграции на другие системы;
   - Необходимость постоянно покупать обновления;
   - Монополизация рынка приводит к завышенным ценам на услуги.

5. **Технические ограничения**
   - Устаревшая клиент-серверная архитектура;
   - Неудобная работа через веб-интерфейс;
   - Сложности с доступом с мобильных устройств;
   - Требовательность к ресурсам компьютера.

**Вывод:**
1С: Управление торговлей - это решение корпоративного уровня, которое экономически нецелесообразно для малых торговых предприятий. Высокие затраты на лицензии, внедрение и поддержку делают его недоступным для небольших продуктовых магазинов с ограниченным бюджетом.

### 1.2.3. Онлайн-сервисы (МойСклад, Класс365, Poster POS)

**Описание:**
Облачные сервисы для автоматизации торговли, работающие по модели SaaS (Software as a Service). Доступны через браузер, оплата в виде ежемесячной подписки.

**Преимущества:**

1. **Быстрый старт**
   - Регистрация и начало работы за 10-15 минут;
   - Не требуется установка программного обеспечения;
   - Готовые шаблоны для разных видов бизнеса.

2. **Доступность**
   - Работа с любого устройства через браузер;
   - Автоматическая синхронизация данных;
   - Мобильные приложения для iOS и Android.

3. **Облачное хранение**
   - Не нужен собственный сервер;
   - Автоматическое резервное копирование;
   - Защита от потери данных.

**Недостатки:**

1. **Ежемесячная абонентская плата**
   - От 2 000 до 15 000 тенге/месяц в зависимости от тарифа;
   - При прекращении оплаты доступ к данным блокируется;
   - Годовые расходы сопоставимы с покупкой собственного решения.

2. **Зависимость от интернета**
   - Без интернета система недоступна;
   - Медленный интернет замедляет работу;
   - Риски при сбоях у провайдера.

3. **Ограничения бесплатных тарифов**
   - Малое количество товаров и операций в месяц;
   - Отсутствие важных функций (аналитика, экспорт данных);
   - Для полноценной работы нужны платные тарифы.

4. **Конфиденциальность данных**
   - Данные хранятся на серверах компании-разработчика;
   - Риски утечки коммерческой информации;
   - Зависимость от политики компании.

**Вывод:**
Онлайн-сервисы удобны для временного использования или тестирования, но создают долгосрочную финансовую зависимость. Для постоянной работы выгоднее иметь собственное решение.

**Обоснование целесообразности:**

После анализа существующих решений можно сделать вывод, что для малого продуктового магазина оптимальным является разработка собственного специализированного веб-приложения.

**Преимущества собственного решения:**

1. **Экономическая эффективность**
   - **Разовые затраты на разработку**: 0 тенге (разработка собственными силами);
   - **Отсутствие ежемесячных платежей**: экономия 24 000 - 180 000 тенге в год по сравнению с облачными сервисами;
   - **Отсутствие лицензионных платежей**: экономия 500 000 - 1 000 000 тенге по сравнению с 1С;
   - **Окупаемость**: система окупается сразу, так как затраты только на разработку.

2. **Точное соответствие требованиям бизнеса**
   - Система содержит только необходимые функции, без избыточности;
   - Интерфейс адаптирован под конкретные бизнес-процессы магазина;

3. **Простота использования**
   - Минималистичный дизайн без перегрузки функциями;
   - Обучение персонала занимает 1-2 часа вместо нескольких дней;
   - Интуитивно понятный интерфейс не требует технических знаний.

4. **Полный контроль над системой**
   - Данные хранятся на собственном сервере;
   - Независимость от внешних поставщиков услуг;

   - Отсутствие риска блокировки доступа из-за неуплаты подписки.

5. **Современные технологии**
   - Использование актуальных веб-технологий (React, FastAPI);
   - Адаптивный дизайн для работы с любых устройств;
   - Быстродействие и оптимизация под конкретные задачи.

6. **Масштабируемость**
   - Возможность добавления новых функций по мере роста бизнеса;
   - Интеграция с дополнительными сервисами (оплата, доставка);

## 1.3. Выбор и обоснование средств реализации проекта

### 1.3.1. Выбор архитектуры приложения

Для реализации веб-приложения выбрана **клиент-серверная архитектура с разделением на frontend и backend** (архитектура SPA - Single Page Application).

**Обоснование:**

1. **Разделение ответственности** - frontend отвечает за пользовательский интерфейс, backend - за бизнес-логику и работу с данными;
2. **Независимая разработка** - команды могут работать над frontend и backend параллельно;
3. **Масштабируемость** - возможность замены или модернизации отдельных компонентов;
4. **Переиспользование** - API backend может использоваться не только веб-интерфейсом, но и мобильными приложениями;
5. **Производительность** - клиентская часть загружается один раз, далее обмен только данными через API.

### 1.3.2. Выбор технологий Backend-разработки

#### Язык программирования: Python 3.11+

**Обоснование выбора Python:**

1. **Простота и читаемость кода**
   - Синтаксис Python близок к естественному языку;
   - Быстрое написание кода - разработка занимает в 2-3 раза меньше времени, чем на Java или C#;
   - Легкость сопровождения и доработки кода.

2. **Богатая экосистема библиотек**
   - Более 300 000 готовых библиотек в репозитории PyPI;
   - Библиотеки для работы с базами данных (SQLAlchemy), веб-фреймворки (FastAPI, Django), обработка изображений (Pillow), генерация PDF (ReportLab);
   - Не нужно писать код с нуля, используются проверенные решения.

3. **Высокая производительность для веб-приложений**
   - Асинхронное программирование (async/await) обеспечивает обработку тысяч запросов в секунду;
   - Оптимизированные библиотеки написаны на C, что дает высокую скорость;
   - Достаточная производительность для нагрузки малого и среднего бизнеса.

4. **Популярность и поддержка**
   - Python входит в топ-3 языков программирования в мире (по индексу TIOBE 2025);
   - Активное комьюнити, множество обучающих материалов;
   - Востребованность на рынке труда - легко найти разработчика для доработки.

5. **Кроссплатформенность**
   - Код работает на Windows, Linux, macOS без изменений;
   - Упрощает развертывание на разных серверах.

**Альтернативы и причины отказа:**
- **JavaScript (Node.js)**: менее читаемый код, слабая типизация может привести к ошибкам;
- **Java**: избыточная сложность, требует больше кода для простых задач;
- **PHP**: устаревающий язык, меньше современных возможностей для асинхронности;
- **C#**: привязка к экосистеме Microsoft, сложности с развертыванием на Linux.

#### Веб-фреймворк: FastAPI 0.109.0

**Обоснование выбора FastAPI:**

1. **Высокая производительность**
   - FastAPI построен на Starlette и Pydantic - одних из самых быстрых компонентов Python;
   - Производительность сравнима с Node.js и Go;
   - Встроенная поддержка асинхронности (async/await) для обработки множества запросов одновременно;
   - **Скорость обработки запросов**: до 20 000 запросов/сек на обычном сервере.

2. **Автоматическая генерация документации**
   - FastAPI автоматически создает интерактивную документацию API в формате Swagger/OpenAPI;
   - Доступ к документации по адресу `/docs` - можно сразу тестировать endpoints;
   - Экономия времени на написание документации вручную.

3. **Автоматическая валидация данных**
   - Использование Pydantic для описания структуры данных;
   - Автоматическая проверка типов, обязательных полей, формата email и других параметров;
   - Защита от некорректных данных на уровне фреймворка.

4. **Современный подход к разработке**
   - Нативная поддержка type hints (аннотация типов) для улучшения читаемости кода;
   - Встроенная поддержка WebSocket для real-time уведомлений;
   - Dependency Injection для удобного управления зависимостями.

5. **Простота освоения**
   - Минимальное количество "магии" - понятная структура кода;
   - Отличная документация с примерами;
   - Активное сообщество разработчиков.

**Сравнение с Django:**

| Критерий | Django | FastAPI |
|----------|--------|---------|
| Скорость работы | Средняя (синхронный) | Очень высокая (асинхронный) |
| Размер фреймворка | Большой (batteries included) | Легковесный (микрофреймворк) |
| Документация API | Требует DRF + drf-yasg | Автоматическая (Swagger) |
| Обучаемость | Сложнее (много концепций) | Проще (минималистичный) |
| Валидация данных | Django Forms/Serializers | Pydantic (автоматическая) |
| Современность | Создан в 2005 году | Создан в 2018 году |
| Подходит для | Монолитные приложения | API-first приложения |

**Вывод:** FastAPI выбран как современный, быстрый и простой фреймворк, идеально подходящий для создания REST API. Django избыточен для данного проекта и уступает FastAPI в производительности.

#### ORM (Object-Relational Mapping): SQLAlchemy 2.0

**Обоснование выбора SQLAlchemy:**

1. **Абстракция от СУБД**
   - Один и тот же код работает с SQLite, PostgreSQL, MySQL и другими базами данных;
   - Легкая миграция между СУБД без переписывания запросов;
   - В разработке используется SQLite, в продакшене PostgreSQL.

2. **Безопасность**
   - Защита от SQL-инъекций на уровне библиотеки;
   - Автоматическое экранирование специальных символов в запросах.

3. **Производительность**
   - Lazy loading - связанные данные загружаются только при необходимости;
   - Eager loading (joinedload) для оптимизации запросов;
   - Connection pooling для повторного использования соединений с БД.

4. **Гибкость**
   - Поддержка как декларативного стиля (модели как классы), так и Core API (сырые SQL-запросы);
   - Возможность написания сложных запросов с join, group by, подзапросами.

### 1.3.3. Выбор системы управления базами данных (СУБД)

Для проекта используется **две СУБД** в зависимости от окружения:

#### SQLite (для разработки и тестирования)

**Обоснование:**

1. **Простота настройки**
   - Не требует установки сервера БД;
   - База данных хранится в одном файле `ecommerce.db`;
   - Мгновенный старт разработки без настройки.

2. **Достаточная производительность**
   - Для небольших нагрузок (до 100 запросов/сек) SQLite работает отлично;
   - Подходит для тестирования и разработки.

3. **Портативность**
   - Легко копировать и переносить базу данных между компьютерами;
   - Удобно для демонстрации проекта.

**Ограничения:**
- Не поддерживает одновременную запись от нескольких пользователей;
- Не подходит для высоконагруженных систем;
- Используется только на этапе разработки.

---

#### PostgreSQL (для production)

**Обоснование:**

1. **Надежность и стабильность**
   - PostgreSQL - промышленная СУБД, используемая крупнейшими компаниями мира;
   - ACID-совместимость гарантирует целостность данных;
   - Устойчивость к сбоям и потере данных.

2. **Производительность**
   - Эффективная обработка тысяч одновременных подключений;
   - Оптимизатор запросов автоматически выбирает наилучший план выполнения;
   - Поддержка индексов, партиционирования, репликации.

3. **Многопользовательский доступ**
   - MVCC (Multi-Version Concurrency Control) позволяет читать и писать данные одновременно без блокировок;
   - Критично для работы нескольких продавцов и администратора одновременно.

4. **Расширенные возможности**
   - Поддержка JSON для хранения неструктурированных данных;
   - Full-text search для поиска по товарам;
   - Триггеры и хранимые процедуры для сложной бизнес-логики.

5. **Бесплатность и open-source**
   - PostgreSQL полностью бесплатна;
   - Нет лицензионных ограничений;
   - Активное сообщество и регулярные обновления.

**Альтернативы:**
- **MySQL**: уступает PostgreSQL в соблюдении стандартов SQL и надежности;
- **MongoDB (NoSQL)**: неподходящая для реляционных данных (товары, заказы, пользователи);
- **Oracle, MS SQL**: коммерческие СУБД с высокой стоимостью лицензий.

### 1.3.4. Выбор технологий Frontend-разработки

#### JavaScript библиотека: React 18

**Обоснование выбора React:**

1. **Популярность и востребованность**
   - React - самая популярная библиотека для создания интерфейсов (по данным Stack Overflow 2025);
   - Используется Facebook, Instagram, Netflix, Airbnb и тысячами других компаний;
   - Огромное сообщество разработчиков и множество готовых решений.

2. **Компонентный подход**
   - Интерфейс разбивается на независимые переиспользуемые компоненты;
   - Легче поддерживать и тестировать код;
   - Ускорение разработки за счет переиспользования компонентов.

3. **Виртуальный DOM**
   - React обновляет только измененные части интерфейса, а не всю страницу;
   - Высокая производительность даже при работе с большими списками товаров;
   - Плавная работа интерфейса без "подвисаний".

4. **Богатая экосистема**
   - Тысячи готовых библиотек компонентов (Material-UI, Ant Design);
   - Инструменты для управления состоянием (Redux, MobX);
   - Библиотеки для работы с формами, графиками, таблицами.

5. **React Hooks**
   - Современный подход к написанию компонентов без классов;
   - Упрощение кода и улучшение читаемости;
   - useState, useEffect для управления состоянием и побочными эффектами.

**Альтернативы:**
- **Vue.js**: менее распространен, меньше готовых решений;
- **Angular**: избыточно сложный для небольших проектов;
- **Vanilla JS**: требует написания большого объема кода вручную.

---

#### Язык: TypeScript 5.2

**Обоснование выбора TypeScript:**

1. **Статическая типизация**
   - Ошибки типов обнаруживаются на этапе разработки, а не в production;
   - Автодополнение кода в IDE ускоряет разработку;
   - Меньше багов - надежнее приложение.

2. **Улучшенная читаемость кода**
   - Явное указание типов переменных, параметров функций и возвращаемых значений;
   - Код становится самодокументируемым;
   - Легче понять код при возвращении к проекту через время.

3. **Рефакторинг без страха**
   - При переименовании или изменении структуры данных TypeScript укажет на все места, требующие изменений;
   - Снижение риска забыть обновить часть кода.

4. **Совместимость с JavaScript**
   - TypeScript компилируется в обычный JavaScript;
   - Можно использовать любые JavaScript библиотеки;
   - Постепенная миграция - можно добавлять типы по мере необходимости.

---

#### Управление состоянием: Redux Toolkit

**Обоснование:**

1. **Централизованное хранилище данных**
   - Все данные приложения (товары, корзина, пользователь) в одном месте;
   - Легко отследить изменения состояния;
   - Упрощение отладки через Redux DevTools.

2. **Предсказуемое изменение состояния**
   - Состояние изменяется только через actions и reducers;
   - Невозможно случайно изменить данные из произвольного места;
   - История всех изменений для отладки.

3. **Redux Toolkit - упрощение Redux**
   - Меньше шаблонного кода (boilerplate);
   - Встроенная интеграция с Redux Thunk для асинхронных операций;
   - RTK Query для работы с API.

---

#### CSS-фреймворк: Tailwind CSS

**Обоснование:**

1. **Utility-first подход**
   - Стилизация компонентов прямо в JSX через классы;
   - Не нужно переключаться между файлами HTML и CSS;
   - Быстрая разработка интерфейса.

2. **Адаптивный дизайн из коробки**
   - Префиксы `sm:`, `md:`, `lg:` для разных размеров экрана;
   - Интерфейс автоматически адаптируется под мобильные устройства, планшеты, десктопы.

3. **Оптимизация размера**
   - PurgeCSS удаляет неиспользуемые стили;
   - Итоговый CSS-файл весит всего 10-20 КБ вместо 500 КБ в Bootstrap.

4. **Консистентный дизайн**
   - Готовая палитра цветов, размеров отступов, шрифтов;
   - Интерфейс выглядит единообразно во всех компонентах.

---

#### Сборщик проекта: Vite

**Обоснование:**

1. **Мгновенный запуск dev-сервера**
   - Холодный старт за 1-2 секунды вместо 30-60 секунд в Webpack;
   - Hot Module Replacement (HMR) - изменения отображаются мгновенно без перезагрузки страницы.

2. **Быстрая сборка**
   - Использование esbuild (написан на Go) вместо Babel;
   - Production build за 10-15 секунд вместо минут.

3. **Современность**
   - Нативная поддержка TypeScript, JSX, CSS Modules;
   - Оптимизация для современных браузеров.
